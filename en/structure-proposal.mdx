# Documentation Structure Proposal

## Overview

This proposal reorganizes the SQD documentation with a focus on developer experience, task-oriented navigation, and progressive disclosure of complexity.

## Key Improvements

### 1. **Chain-First Organization**

- Developers typically think "I need to index Ethereum" not "I need to learn about processors"
- Each chain section is self-contained with its own quickstart and examples

### 2. **Progressive Disclosure**

- Start with quickstarts and basic examples
- Advanced topics are clearly separated
- API references are accessible but not front-and-center

### 3. **Task-Oriented Sections**

- "How do I..." structure for common tasks
- Real-world examples prominently featured
- Clear separation between learning and reference material

### 4. **Unified CLI Documentation**

- Single source of truth for all CLI commands
- Workflow-based documentation showing command sequences
- Integration with deployment sections

### 5. **Clear Migration Paths**

- Dedicated migration section
- Step-by-step guides for common migrations
- Before/after code comparisons

## Navigation Principles

1. **Minimize Depth**: No more than 3 levels deep
2. **Predictable Paths**: Similar content in similar locations across chains
3. **Cross-References**: Liberal use of "See also" sections
4. **Search-Optimized**: Page titles that match what developers search for

## Content Guidelines

1. **Start with Why**: Each page explains the problem it solves
2. **Show, Don't Tell**: Code examples before explanations
3. **Complete Examples**: All code snippets should be runnable
4. **Version Everything**: Clear version indicators on all pages
5. **Test Everything**: All examples must be tested before publishing
