## Why Portal Matters

Modern blockchain data engineering faces a fundamental problem: RPC was designed for wallets and transactions, not for data extraction at scale.

### The problem

<Warning>
  - Single wallet history queries can timeout on RPC - Backfills on high-history
  chains (BSC, Solana) routinely break - High-TPS chains push RPC-centric
  designs past their limits - Teams resort to centralized data warehouses with
  vendor lock-in
</Warning>

### Portal's solution

<Check>
  - Query arbitrary block ranges in a single HTTP request - Use native streaming
  for constant memory usage - Rely on built-in finality handling and reorg
  compensation - Deploy anywhere: managed, self-hosted, or decentralized
</Check>

## Core Capabilities

Portal provides four key features that make it ideal for data pipelines.

### 1. Arbitrary block ranges

Request any block range in a single HTTP POST—no manual pagination through individual blocks. Essential for backfills and historical analysis.

```http
POST /stream
fromBlock: 18000000
toBlock: 18100000
```

### 2. Native finalization

Built-in reorg handling with chain-specific finality rules. Portal communicates rollbacks as structured events in the stream so you can replay safely.

<Tip>Use `/finalized-stream` when you need production-safe data ingestion.</Tip>

### 3. HTTP streaming

Newline-delimited JSON (NDJSON) responses enable constant-memory processing of massive ranges. Process terabytes without loading everything into RAM.

### 4. Flexible deployment

Use the same interface across managed endpoints, self-hosted instances, or the decentralized SQD Network. Swap backends without code changes.

<Note>Deploy anywhere, preserve sovereignty, and avoid lock-in.</Note>
